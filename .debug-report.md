## Hypotheses

### H-001: Refresh token expired past the 100-day inactivity window
- **Likelihood:** High
- **Test:** Attempt refresh_token grant against Intuit production token endpoint.
- **Observation:** `400 invalid_grant`. After Playground re-consent + Keychain reseed, refresh succeeds (200) and live P&L returns 200.
- **Outcome:** SUPPORTED

### H-002: Server never recovers because session is built once at boot
- **Likelihood:** High
- **Test:** Reseed Keychain with a valid token, then call a tool on the already-running server.
- **Observation:** Native `mcp__quickbooks__*` still returns "session not initialized" — the process cached `quickbooks = None` from the failed boot; tool bodies only guard `if quickbooks is None` and never re-init.
- **Outcome:** SUPPORTED

### H-003: qb helper broken by upstream removal of the `--mcp-cli` flag
- **Likelihood:** High
- **Test:** Inspect shim; check `claude --help` for `--mcp-cli`.
- **Observation:** Shim execs `claude --mcp-cli "$@"`; flag absent from Claude CLI 2.1.201; `claude mcp` only manages servers (no tool-call subcommand). Fails even on the auth-free schema path.
- **Outcome:** SUPPORTED

### H-004: Modernized Reports API changes the ProfitAndLoss response shape (parser break)
- **Likelihood:** Medium (per ticket concern)
- **Test:** Call ProfitAndLoss with `testing_migration=true` and diff shape/headers vs legacy.
- **Observation:** Shapes identical (same top-level + Header keys + Rows structure); only difference is `v3modernResponse: true`. Server is a pure passthrough (`str(response)`) — no parser to break.
- **Outcome:** FALSIFIED (no shape change; no correctness code change required for reports)

## Root Cause
1. **Session outage:** the Keychain refresh token expired (invalid_grant). Compounded by a **resilience defect** — the server constructs `QuickBooksSession` once at import time and, on failure, leaves `quickbooks = None` permanently; tool bodies only check `if quickbooks is None` and never re-initialize, so even after the Keychain holds a valid token the running process stays dead until manual restart.
2. **qb helper:** `~/.local/bin/mcp-cli` execs `claude --mcp-cli "$@"`; the `--mcp-cli` flag was removed from the Claude CLI and no equivalent tool-call subcommand exists, so the shim is unusable.
3. **Reports modernization:** not a defect — the modernized ProfitAndLoss payload is structurally identical to legacy; the passthrough server is unaffected. ProfitAndLoss is the only report exposed.

## Fix
1. **Auth (operational, DONE):** re-consent via Intuit OAuth 2.0 Playground → reseed Keychain. Verified.
2. **Self-healing session init (code):** introduce a lazy `get_session()` accessor that (re)initializes `QuickBooksSession` on demand when the cached session is missing, so a reseeded/rotated token is picked up without a process restart. Route all tool bodies through it.
3. **qb helper (code):** add an in-repo direct MCP stdio client (`qb_cli.py`) that speaks JSON-RPC to the server over stdio; repoint the `qb` shell function to it — removing the dependency on the removed `mcp-cli`/`--mcp-cli` path.
4. **Docs:** README — re-auth-via-Playground + Keychain reseed procedure, interim QBO-UI manual-export fallback, and the verified "modernized ProfitAndLoss needs no parser change (v3modernResponse=true confirmed)" finding.

## Verification
- Token refresh returns 200 and live ProfitAndLoss returns 200 with intact revenue/COGS/expense/NOI rows (done, above).
- After self-healing change: with `quickbooks` unset, a tool call lazily initializes the session and returns data (no restart).
- `qb Account` (schema) and `qb Account "Name = 'x'"` (query) succeed via the new client.
- Modernized confirmation: ProfitAndLoss call surfaces `v3modernResponse=true` and parses identically.
