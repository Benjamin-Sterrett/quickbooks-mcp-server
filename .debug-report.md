# Debug Report — Security Scan workflow failing every Monday

**Tier:** Incident (production credentials exposed in public repo)
**Started:** 2026-05-18
**ROOT-CAUSE-PROVEN:** true

## Repro

Trigger the scheduled or workflow_dispatch run of `.github/workflows/security.yml` on `benshph-blip/quickbooks-mcp-server` master at `ba801a2`. Every run since 2026-03-02 (12 consecutive) fails.

```
$ gh run list --workflow=security.yml --repo benshph-blip/quickbooks-mcp-server --limit 12 --json conclusion,createdAt
[all 12: "failure"]
```

## Observed Behavior

Two distinct failure modes per run:

1. **`Secrets Detection` job** — gitleaks exits 2 with finding:
   ```
   Finding:     QUICKBOOKS_CLIENT_SECRET=REDACTED
   RuleID:      generic-api-key
   File:        .handoff.md
   Line:        14
   Commit:      1773a302cffe06c7c52b14a04832ff6cfb62a374
   ```
   Plus a second finding at line 15 for `QUICKBOOKS_REFRESH_TOKEN`.

2. **`Create Alert Issue` job** — HTTP 410:
   ```
   ##[error]Unhandled error: HttpError: Issues has been disabled in this repository.
   POST https://api.github.com/repos/.../issues → 410
   ```

## Root Cause

**Proximate cause (job 1):** `.handoff.md` was committed in PR #1 (`1773a30`, 2026-02-23) containing live `.env` contents (CLIENT_SECRET + REFRESH_TOKEN). Three-element proof:
- **Reproduction:** `git log --all -p -S "HuvklLp..." --oneline` returns commit `1773a30` (confirmed before history rewrite).
- **Isolation:** `git filter-repo --invert-paths --path .handoff.md` removes the finding (confirmed: post-rewrite `git log -p -S` returns empty).
- **Completeness:** the only path matching gitleaks' `generic-api-key` rule with entropy > 4.3 in the repo is `.handoff.md:14-15`. IS/IS-NOT confirmed.

**Proximate cause (job 2):** The `notify-on-failure` job uses `actions/github-script` to call `github.rest.issues.create()` unconditionally. This fork has `has_issues: false` (default for forks). The API returns 410.

**Root cause (systemic, both jobs):** Agent session-handoff files (`.handoff.md`, `.auto-resume.md`, etc.) capture working-tree state including environment-variable dumps. They were never added to `.gitignore`. `git add .` swept `.handoff.md` into PR #1 silently. The CI was correctly designed to catch this but its alert mechanism assumed Issues were enabled — assumption invalid on forks.

## Fix

1. **Mitigation (already done, separate commit on master via force-push):** rotate client_secret in Intuit, rotate refresh_token via OAuth, scrub `.handoff.md` from full history via `git filter-repo`, force-push master.
2. **Workflow hardening (this commit):**
   - Add `has_issues` check before issue creation; no-op if disabled
   - Write structured failure summary to `$GITHUB_STEP_SUMMARY` (always works)
   - Optional Telegram alert gated on secret presence
   - All alert sub-steps `continue-on-error: true`
3. **Prevention (this commit):** `.gitignore` blocks all agent artifact patterns.
